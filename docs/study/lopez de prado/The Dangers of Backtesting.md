## 절별 내용과 PPO 그리드 봇 관점 응용

### 11.1 Motivation — 백테스트의 본질

핵심 주장: 백테스트는 물리 실험과 다르다. 환경 변수를 통제하며 반복 가능한 실험이 아니라 가설(hypothetical)일 뿐이며, 아무것도 증명하지 않는다.

**PPO 봇에 적용**: 사용자가 백테스트에서 Sharpe 2.5를 얻었다고 해서, 그게 "이 봇이 미래에 Sharpe 2.5를 낼 것"을 증명하지 않음. 한 번의 시뮬레이션 결과일 뿐. 졸업 논문 결론부에서 "Sharpe 2.5를 달성했다"가 아니라 "이 데이터셋, 이 설정에서 Sharpe 2.5가 관측되었으며 다음과 같은 한계가 있다"로 써야 함.

### 11.2 Mission Impossible: The Flawless Backtest — "7대 죄악"

이 절의 핵심은 백테스트가 빠지기 쉬운 **체계적 오류 목록**. 책 외에도 López de Prado가 강연/논문에서 반복적으로 정리해온 항목들:

1. **Survival bias (생존 편향)** — 살아남은 자산만 데이터에 들어감
2. **Look-ahead bias (전방 정보 누설)** — 미래 정보가 현재 의사결정에 슬쩍 들어감
3. **Storytelling (이야기 짓기)** — 결과를 본 뒤 그럴듯한 가설을 사후에 끼워맞춤
4. **Data mining / Data snooping** — 같은 데이터에 수많은 가설을 시험해 우연한 적중을 발견
5. **Transaction costs** — 비용 모델이 비현실적
6. **Outliers** — 소수의 극단값이 전체 수익을 지배
7. **Shorting** — 차입/공매도 가능성을 과대 가정

**그리드 봇별 매핑**:

|죄악|사용자 프로젝트에서의 발현 형태|위험도|
|---|---|---|
|Survival bias|BTC 단일자산이라 직접 위험은 낮음. 단, "BTC가 2017→2026 사이 살아남았다"는 자체가 메타 선택 편향|중|
|**Look-ahead bias**|`divergence`, `volatility` 계산 시 윈도우에 현재 시점 포함 여부. 정규화 통계를 전체 데이터에서 계산|**상**|
|Storytelling|"PPO가 잘 작동한 이유는 BTC의 mean-reverting 특성 때문" 같은 사후 합리화|중|
|**Data mining**|**시드 변경, 하이퍼파라미터 탐색, 보상함수 v1/v2/v3 반복** — RL에서 가장 위험|**최상**|
|Transaction costs|14.6에서 다뤘듯 그리드 봇은 거래 빈도 높아 치명적|상|
|Outliers|BTC는 fat-tail. 2020년 3월, 2021년 5월 같은 단일 사건이 전체 결과를 결정|상|
|Shorting|현재 설계가 long-only 그리드라면 무관|하|

### 11.3 Even If Your Backtest Is Flawless, It Is Probably Wrong — **이 챕터의 핵심**

설령 위 7가지를 모두 피했다 해도, **선택 편향(selection bias)이 남아 있음**. 이게 RL에서 가장 큰 함정:

> N번 시도해서 최고 결과를 보고하면, 그 결과는 통계적으로 거의 확실히 위로 편향됨.

PPO 학습의 현실:

- 시드 10개 돌려서 가장 잘 된 것 보고
- 학습률 5개 시도해서 가장 잘 된 것 채택
- 보상함수 3가지 변형 중 가장 잘 된 것 선택
- 평가 구간 여러 개 중 잘 나온 것 강조

이게 누적되면 **실제 일반화 성능과 보고 성능의 괴리가 폭발적으로 커짐.** 1.0 Sharpe짜리 노이즈에서 N번 시도하면 2.0 Sharpe짜리 "성과"가 우연히 나옴.

**대응**: López de Prado가 이를 정량화하는 도구로 제안하는 게 **PBO (Probability of Backtest Overfitting)** 와 **Deflated Sharpe Ratio (DSR)** — 14장 14.4에서 다룬 그 DSR. 11장과 14장이 여기서 연결됨.

### 11.4 Backtesting Is Not a Research Tool — 가장 도발적인 절

저자의 주장: **백테스트로 전략을 "발견"하지 말라.** 백테스트는 이미 _이론적으로 도출된_ 전략을 사후 검증하는 도구일 뿐, 백테스트 결과를 보고 전략을 수정하는 순간 selection bias가 시작됨.

**RL 프로젝트의 곤란한 점**: PPO의 본질이 "환경에서 시행착오로 학습"이므로 이 원칙과 정면 충돌. 사용자 프로젝트는 구조적으로 백테스트(환경 시뮬레이터)를 연구 도구로 사용함.

**현실적 타협안 (졸업 논문에 명시할 것)**:

- **Train / Validation / Test 시기적 분리**를 엄격히. 예: 2017~2022 train, 2023 validation (하이퍼파라미터 선택), 2024~2026 test (한 번만 평가).
- Test set 결과는 **딱 한 번만 본다.** 재학습 금지.
- 만약 test 성과가 나쁘다면 그게 진실. 다시 train으로 돌아가 튜닝하면 test set이 더 이상 test가 아니게 됨.

이 규율을 졸업 논문 methodology 절에 명시하면 심사 방어력이 크게 상승.

### 11.5 A Few General Recommendations

López de Prado의 처방:

- **여러 시나리오에서 평가** (서로 다른 시장 레짐, 자산, 구간)
- **이론적 가설을 먼저, 백테스트는 나중에**
- **결과의 통계적 유의성을 명시적으로 평가** (PSR, DSR, PBO)
- **민감도 분석** — 하이퍼파라미터 작은 변화에 결과가 얼마나 흔들리는가
- **연구 일지 기록** — 몇 번 시도했는지 트래킹해야 DSR에 N을 넣을 수 있음

**사용자에게 가장 실용적인 조언**: **모든 시드/하이퍼파라미터 시도 횟수를 기록하는 로그를 만들라.** PPO 학습 스크립트가 자동으로 N(누적 시도 횟수)을 카운트하게 해두면, 졸업 논문에서 DSR을 계산할 때 그대로 사용 가능. 보통 RL 학생들이 이걸 놓침.

### 11.6 Strategy Selection — 다중 비교 문제의 정량화

이 절은 **K개 전략 후보 중 최고를 고를 때 어떻게 그 선택의 통계적 인플레이션을 보정할 것인가**를 다룸. 위 11.3의 형식적 해결.

**PPO 봇에 적용**:

- 보상함수 변형 K=3, 시드 M=10이면 사실상 30개 "전략"을 시험
    
- 그중 최고 Sharpe를 보고할 때 **Bonferroni 보정 또는 DSR**로 deflate해야 함
    
- López de Prado의 공식:
    
    $$\text{DSR} = \text{PSR}(SR_0)$$
    
    where $SR_0 = E[\max SR_n]$ 는 N번 시도 시 최대 Sharpe의 기댓값. N이 클수록 임계값이 높아짐.
    

졸업 논문에 들어갈 결과 형식:

> "최고 시드 기준 Sharpe 2.1, 10개 시드 평균 1.4 ± 0.5, DSR = 0.62 (목표 Sharpe 0 대비 통계적 유의성 확률 62%)"

이렇게 쓰면 심사위원이 selection bias를 의심할 여지가 줄어듦.

## 졸업 프로젝트 관점 권장 활용법

1. **이번 주 (보상함수 설계 단계)에는 11장을 정독할 필요 없음.** 한 번 훑고 12, 14, 15장에 시간 투입.
2. **백테스트 코드를 짜기 직전에 11장 다시 읽기.** 7대 죄악 체크리스트로 시뮬레이터 코드를 점검.
3. **실험 시작 직전 다음 3가지를 코드로 박아둘 것**:
    - Train/Val/Test 시기 분리 (재학습 금지 규칙을 코드 레벨에서 강제)
    - 시도 횟수 자동 로깅 (N for DSR)
    - 거래 비용/슬리피지 모델을 환경에 하드코딩
4. **논문 작성 단계에서 11장의 7대 죄악을 limitation 절의 구조로 활용.** "본 연구는 다음과 같이 각 위험에 대응했다 / 또는 다음과 같은 한계로 남았다" 형식. 심사위원이 가장 좋아하는 정직성.

## 한 가지 짚어둘 점

11장은 **지도학습 전략 가정**에서 쓰여, RL에 그대로 매핑되지 않는 부분이 있음. 특히 11.4의 "백테스트로 발견하지 말라"는 원칙은 RL의 학습 메커니즘 자체와 충돌. **이걸 졸업 논문에서 정직하게 인정하고 어떻게 완화했는지 서술하는 게 가장 우아한 처리.** 예:

> "본 연구의 PPO 에이전트는 환경 상호작용을 통해 정책을 학습하므로 López de Prado(2018)가 경고한 백테스트-as-research-tool 패러다임을 구조적으로 따른다. 이를 완화하기 위해 (1) train/val/test 시기 분리를 엄격히 적용했으며, (2) test 구간 결과는 단일 평가로 보고하고 재학습을 수행하지 않았으며, (3) Deflated Sharpe Ratio를 보고해 다중 시도의 선택 편향을 정량적으로 보정했다."

이 한 문단이 졸업 심사에서 "RL 백테스트의 한계 인식" 질문에 대한 완벽한 방어가 됨.
